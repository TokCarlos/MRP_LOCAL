Abaixo está a lista consolidada para virar regra absoluta do projeto **MRP_LOCAL / Sistema Local**.

# REGRAS ABSOLUTAS — MRP_LOCAL

## 1. Regra central do projeto

1. O sistema é **local-first**.

2. O sistema deve funcionar em rede local, sem depender obrigatoriamente de hospedagem externa.

3. O ambiente atual é **teste/simulação**, rodando no servidor/PC de casa.

4. A raiz de teste atual é `X:\`.

5. A estrutura atual em `X:\` deve ser preservada como padrão de desenvolvimento.

6. A arquitetura deve permitir migração futura para servidor real da empresa sem reescrever lógica.

7. A raiz física deve ser configurável.

8. Nenhuma lógica de negócio pode depender diretamente de `X:\`.

9. Nenhuma lógica de negócio pode depender diretamente de `\\HOME-MACHINE`.

10. Nenhuma lógica de negócio pode depender diretamente de usuário Windows específico.

11. Nenhuma lógica de negócio pode depender diretamente de letra de unidade mapeada.

12. Caminhos fixos só são aceitos em arquivos de configuração, scripts de ambiente ou documentação operacional.

13. O sistema deve ser portátil/configurável.

14. O sistema não deve depender de MEGA.

15. MEGA foi descartado como estratégia do projeto.

16. Acesso remoto, quando necessário, deve seguir a ideia de **Tailscale + rede local segura**, não sincronização por nuvem.

17. O sistema deve ser capaz de rodar automaticamente no servidor/PC de teste.

18. O objetivo final é um sistema local estável, acessível pela rede, com frontend web e backend/local services.

---

## 2. Regra de metodologia

19. Antes de codar, documentar.

20. Antes de alterar código, identificar problema, causa, solução e risco.

21. Nenhuma alteração deve ser feita “no escuro”.

22. Implementações já corrigidas e funcionais não devem ser alteradas sem necessidade técnica real.

23. Se algo funcional precisar ser tocado por integração, isso deve ser registrado.

24. Toda alteração deve ter versão.

25. Toda alteração deve ter escopo.

26. Toda alteração deve ter histórico.

27. Toda alteração deve ter status.

28. Toda alteração deve ter problema e solução documentados.

29. O sistema deve ser trabalhado por módulos.

30. O sistema deve ser trabalhado por etapas.

31. O sistema deve ter registro claro do que já foi concluído.

32. O sistema deve ter registro claro do que ainda está pendente.

33. Não chamar nada de “final”, “blindado” ou “homologado” sem validação real.

34. Separar sempre: validado localmente, validado pelo usuário, parcial, pendente ou não homologado.

35. Não acumular gambiarras silenciosas.

36. Não corrigir uma falha criando outra camada obscura.

37. Correção boa é rastreável, documentada e reversível.

38. Todo patch deve preservar o que já está certo.

39. A lógica correta deve ser consolidada em documentação para uso futuro em Codex/Projetos.
